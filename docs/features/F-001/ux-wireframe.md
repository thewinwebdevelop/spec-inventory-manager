## 1. Flow map (ภาพรวม)

```
[Landing/Marketing] ──(สมัครใช้งาน)──▶ [Signup] ──สำเร็จ──▶ [Login (email เติมไว้)]
                                          │                        │
                                    (มีบัญชีแล้ว?)                 │ สำเร็จ
                                     "เข้าสู่ระบบ" ◀───────────────┤
                                          ▲                        ▼
                                   [Login] ◀──throttle/ผิด──   [Org context: F-002]
                                          │
                                    (ลืมรหัสผ่าน?)
                                          ▼
                          [ขอความช่วยเหลือ — ติดต่อ Owner/Admin]
                          (self-serve reset = F-081, ยังไม่มีในหน้านี้)

[ตั้งค่าบัญชี] ──▶ ["ความปลอดภัย" /settings/security] ──▶ [เปลี่ยนรหัสผ่าน (US-6)] ──สำเร็จ──▶ toast
                                                  │            "อุปกรณ์อื่นถูกออกจากระบบ" (§9)
                                                  └──▶ [รายการอุปกรณ์ที่ล็อกอินอยู่] ──▶ ออกจากอุปกรณ์นี้ (รายตัว)
                                                                                     └─▶ ออกจากระบบทุกอุปกรณ์ (มี confirm)

[Any authenticated screen] ──token หมดอายุ (เงียบ)──▶ ต่ออายุอัตโนมัติ (ไม่รบกวนผู้ใช้)
                            ──refresh ใช้ไม่ได้/ถูกเพิกถอน──▶ เด้งออกไปหน้า Login + toast สุภาพ
```

## 2. Signup flow (US-1)

### หน้าจอ: "สมัครใช้งาน" (`/signup`)

**องค์ประกอบ**
- Logo/brand header
- ฟิลด์: อีเมล (type=email), รหัสผ่าน (type=password, มีปุ่มตา "แสดง/ซ่อนรหัสผ่าน")
- ตัวช่วยแสดงเกณฑ์รหัสผ่าน (helper text ใต้ฟิลด์ ไม่ใช่ popup ที่หายไป): "อย่างน้อย 8 ตัวอักษร"
- ปุ่มหลัก: "สมัครใช้งาน" (primary, full-width บนมือถือ)
- ลิงก์รอง: "มีบัญชีแล้ว? เข้าสู่ระบบ" → `/login`

**Copy (ไทย)**
| องค์ประกอบ | ข้อความ |
|---|---|
| หัวข้อ | สมัครใช้งาน OmniStock |
| คำอธิบายรอง | เริ่มต้นจัดการสต๊อกและบัญชีร้านค้าของคุณ |
| label อีเมล | อีเมล |
| placeholder อีเมล | เช่น somchai@shop.com |
| label รหัสผ่าน | รหัสผ่าน |
| placeholder รหัสผ่าน | อย่างน้อย 8 ตัวอักษร |
| helper รหัสผ่าน (สถานะปกติ) | ใช้ตัวอักษร ตัวเลข หรือวลีที่จำง่าย — ไม่ต้องมีอักขระพิเศษก็ได้ |
| ปุ่มหลัก | สมัครใช้งาน |
| ปุ่มหลัก (loading) | กำลังสมัคร... |
| ลิงก์ไปหน้า login | มีบัญชีอยู่แล้ว? เข้าสู่ระบบ |

**Validation states (inline, ไม่ใช่ modal)**

| สถานการณ์ | ทริกเกอร์ | UI | ข้อความ |
|---|---|---|---|
| อีเมลรูปแบบผิด | blur ฟิลด์ / submit | border แดง + ข้อความใต้ฟิลด์ | "รูปแบบอีเมลไม่ถูกต้อง" |
| รหัสผ่านสั้นเกิน (`422 PASSWORD_TOO_SHORT`) | submit → server ตอบ | border แดง + ข้อความใต้ฟิลด์รหัสผ่าน, focus กลับไปที่ฟิลด์ | "รหัสผ่านสั้นเกินไป ต้องมีอย่างน้อย 8 ตัวอักษร" |
| รหัสผ่านอยู่ในลิสต์รั่ว/ยอดฮิต (`422 PASSWORD_BREACHED`) | submit → server ตอบ | border แดง + ข้อความใต้ฟิลด์ | "รหัสผ่านนี้ถูกใช้งานทั่วไปมาก ไม่ปลอดภัย ลองตั้งรหัสผ่านที่คาดเดายากขึ้น" |
| อีเมลซ้ำ (`409 EMAIL_TAKEN`) | submit → server ตอบ | border แดงที่ฟิลด์อีเมล + ข้อความใต้ฟิลด์ + ลิงก์ช่วยเหลือ | "อีเมลนี้มีผู้ใช้งานแล้ว" + ลิงก์ inline "เข้าสู่ระบบ" ต่อท้ายประโยค |
| Rate limited (`429`, IP throttle) | submit → server ตอบ | banner เหนือฟอร์ม (ไม่ใช่ toast ที่หาย) + ปุ่ม submit disabled จนกว่านับถอยหลังจะครบ | ดู §4 (ใช้ copy เดียวกับ throttle countdown) |
| Network/5xx | submit ล้มเหลว | banner แดงเหนือฟอร์ม + ปุ่ม "ลองใหม่" | "เกิดข้อผิดพลาด กรุณาลองใหม่อีกครั้ง" |
| กำลังส่ง | submit กด | ปุ่มเป็น loading state, disabled ป้องกันกดซ้ำ | "กำลังสมัคร..." |
| สำเร็จ (`201`) | server ตอบ 201 | redirect ไป `/login?email=<ที่กรอกไว้>` พร้อม toast สีเขียวชั่วคราว | "สมัครสำเร็จ! เข้าสู่ระบบเพื่อเริ่มใช้งาน" |

**หมายเหตุความเป็นมิตร (friendliness):** ข้อความ error ทุกอันบอก "ทำไม" และ "ทำอะไรต่อ" ไม่ใช่แค่ชื่อ error code — เช่นไม่ใช้ "PASSWORD_BREACHED" ตรงๆ แต่แปลเป็นเหตุผลที่แม่ค้าเข้าใจ ("ถูกใช้งานทั่วไปมาก ไม่ปลอดภัย").

**สิ่งที่จงใจไม่ทำ:** ไม่มี "ยืนยันรหัสผ่าน" ฟิลด์ที่สอง (ลดแรงเสียดทาน, ใช้ปุ่มแสดง/ซ่อนรหัสผ่านแทนเพื่อให้ตรวจทานได้) — ปุ่มตาแสดงรหัสผ่านทำหน้าที่แทน.

## 3. Login flow (US-2, US-5 throttle)

### หน้าจอ: "เข้าสู่ระบบ" (`/login`)

**องค์ประกอบ**
- ฟิลด์: อีเมล (เติมล่วงหน้าได้จาก query param หลังสมัคร), รหัสผ่าน (ปุ่มตาแสดง/ซ่อน)
- ปุ่มหลัก: "เข้าสู่ระบบ"
- ลิงก์: "ลืมรหัสผ่าน?" → นำไปหน้า/section "ขอความช่วยเหลือ" (ดู §3.3 — ไม่ใช่ self-serve reset ใน MVP)
- ลิงก์: "ยังไม่มีบัญชี? สมัครใช้งาน" → `/signup`

**Copy (ไทย)**
| องค์ประกอบ | ข้อความ |
|---|---|
| หัวข้อ | เข้าสู่ระบบ |
| label อีเมล | อีเมล |
| label รหัสผ่าน | รหัสผ่าน |
| ปุ่มหลัก | เข้าสู่ระบบ |
| ปุ่มหลัก (loading) | กำลังเข้าสู่ระบบ... |
| ลืมรหัสผ่าน | ลืมรหัสผ่าน? |
| ลิงก์ไป signup | ยังไม่มีบัญชี? สมัครใช้งาน |

### 3.1 Error ทั่วไป (enumeration-safe — `401 INVALID_CREDENTIALS`)

**ต้องแสดงข้อความเดียวกันทุกกรณี** (รหัสผิด / อีเมลไม่มีในระบบ / กำลังถูก throttle แบบยังไม่ถึง `429`):

> **"อีเมลหรือรหัสผ่านไม่ถูกต้อง"**

- ไม่ระบุว่าอีเมลหรือรหัสผ่านผิดข้อไหน (ตาม AC US-2 + arch §9)
- แสดงเป็น banner สีแดงเหนือฟอร์ม (ไม่ใช่ toast ลอย — ผู้ใช้ต้องเห็นชัดว่าต้องแก้อะไร) ทั้งฟิลด์อีเมล+รหัสผ่าน **border แดงพร้อมกันทั้งคู่** (จะไม่ไฮไลต์แค่ฟิลด์เดียว เพราะจะกลายเป็นการรั่วว่า "ผิดที่ฟิลด์นี้")
- โฟกัสกลับไปที่ฟิลด์รหัสผ่าน (เคลียร์ค่ารหัสผ่านเดิมทิ้ง, อีเมลเก็บไว้ให้ไม่ต้องพิมพ์ซ้ำ)

### 3.2 Throttle countdown (`429 RATE_LIMITED`, D-005 — ต้องมี, ห้ามให้เดา)

เมื่อ server ตอบ `429` พร้อม header `Retry-After: <วินาที>`:

- **ห้ามใช้คำว่า "บัญชีถูกล็อก" / "ถูกระงับ" เด็ดขาด** (D-005 — ผิดหลัก anti-lockout, ทำให้ผู้ใช้ตื่นตระหนกเกินจริงว่าเข้าไม่ได้อีกแล้ว)
- แสดงเป็น **banner เหนือฟอร์ม** แทน error แดงปกติ (สีเหลือง/นิ่วเตือน ไม่ใช่สีแดงอันตราย — สื่อว่า "รอได้ ไม่ใช่หายนะ")
- ฟอร์ม (ฟิลด์ + ปุ่ม submit) **disabled** ระหว่างนับถอยหลัง กันผู้ใช้กดรัวๆ ทำให้งง
- นับถอยหลังเป็น **นาที:วินาที ที่ปรับ real-time ทุกวินาที** ไม่ใช่เลขนิ่ง (ผู้ใช้ต้องเห็นว่ามันกำลังลดลงจริง = อุ่นใจว่าไม่ค้าง)
- เมื่อนับถึง 0 → **เปิดฟอร์มให้กดใหม่อัตโนมัติ** (ไม่ต้อง refresh หน้า)

**Copy:**
| สถานะ | ข้อความ |
|---|---|
| เริ่มนับถอยหลัง (>60 วิ) | "ลองเข้าสู่ระบบถี่เกินไป กรุณารอสักครู่แล้วลองใหม่ · เหลือ {mm}:{ss} นาที" |
| นับถอยหลัง (≤60 วิ) | "รอสักครู่แล้วลองใหม่ · เหลือ {N} วินาที" |
| นับถึง 0 | banner หายไปเอง ฟอร์ม enable กลับ, ไม่ต้องมีข้อความเพิ่ม |
| helper เสริม (บรรทัดเล็กใต้ banner) | "เพื่อความปลอดภัยของบัญชีคุณ ระบบจะให้ลองใหม่ได้เองเมื่อครบเวลา" |

> เหตุผลของ helper เสริม: ปิด anxiety ของ solo owner (D-005 gap) — สื่อชัดว่า "นี่คือกลไกความปลอดภัยที่หายเองได้" ไม่ใช่ "แอดมินแบนคุณ" ผู้ใช้จึงไม่ต้องเดาว่าต้องรอใคร action ให้.

**คำนวณ mm:ss:** client รับ `Retry-After` (วินาที) ตอน response มา → ตั้ง local countdown timer นับถอยหลังเอง (ไม่ poll ซ้ำทุกวินาที) เผื่อ clock skew เล็กน้อยด้วยการ re-sync ทุกครั้งที่ผู้ใช้พยายาม submit ซ้ำระหว่างรอ (server จะตอบ `429` + `Retry-After` ใหม่เสมอถ้ายังไม่ครบเวลา — ใช้ค่านั้นทับ countdown เดิม).

### 3.3 "ลืมรหัสผ่าน?" ใน MVP (ไม่มี self-serve reset — F-081 ทำทีหลัง)

กดแล้วไม่ใช่ฟอร์ม "กรอกอีเมลเพื่อรับลิงก์" (ยังไม่มี SMTP) แต่เป็นข้อความช่วยเหลือแบบ static:

> **หัวข้อ:** ลืมรหัสผ่าน?
> **เนื้อหา:** "ตอนนี้ระบบยังไม่รองรับการขอรีเซ็ตรหัสผ่านด้วยตัวเอง กรุณาติดต่อเจ้าของร้าน/ผู้ดูแลระบบ (Owner/Admin) ในทีมของคุณ เพื่อให้ตั้งรหัสผ่านใหม่ให้"
> **ปุ่ม:** "กลับไปเข้าสู่ระบบ"

**Solo-owner edge (D-005):** ถ้าไม่มี Owner/Admin คนอื่น (ลืมรหัสผ่านตัวเอง คนเดียวในองค์กร) ให้เพิ่มบรรทัดต่อท้าย:

> "หากคุณเป็นเจ้าของร้านเพียงคนเดียวและลืมรหัสผ่าน กรุณารอสักครู่แล้วลองพิมพ์รหัสผ่านที่คิดว่าถูกต้องอีกครั้ง (ระบบจะไม่ล็อกบัญชีคุณถาวร)"

> หน้านี้เป็น **static/FAQ-style**, ไม่ผูกกับ API ใดๆ — ไม่ต้องรู้ว่าผู้ใช้เป็น solo owner จริงไหม (ไม่มี endpoint ให้เช็ค และการเช็คจะเป็นการรั่วข้อมูล org) จึงเขียน copy แบบ "if" ให้ครอบคลุมทั้งสองเคส.

## 4. Multi-device session list (US-3)

### หน้าจอ: "อุปกรณ์ที่เข้าสู่ระบบ" (`/settings/sessions` หรือใต้ "บัญชีของฉัน")

**Layout:** 1 row ต่อ 1 family (device session) เรียงจาก `lastUsedAt` ล่าสุดก่อน

**แต่ละ row แสดง**
- ไอคอนอุปกรณ์ (จำแนกจาก `deviceId`/UA อย่างหยาบ: มือถือ / เดสก์ท็อป — ไม่ต้องแม่นยำ 100%)
- label: "อุปกรณ์นี้" (ถ้า `current: true`) หรือชื่อที่อ่านง่ายกว่า raw `deviceId` (เช่น "เบราว์เซอร์ · Chrome" ถ้า parse ได้ ไม่งั้น fallback "อุปกรณ์ไม่ทราบชื่อ")
- "ใช้งานล่าสุด: {relative time เช่น 'เมื่อ 2 นาทีที่แล้ว' / 'เมื่อวาน 14:32'}" จาก `lastUsedAt`
- badge "อุปกรณ์นี้" (สีเขียวอ่อน, เด่นแต่ไม่ก้าวร้าว) เฉพาะ row ที่ `current: true`
- ปุ่ม action: แถวที่ `current: false` → "ออกจากอุปกรณ์นี้" (secondary/outline button, สีเทา ไม่ใช่สีแดง — เป็น action ปกติไม่ใช่ destructive ร้ายแรง); แถวที่ `current: true` → **ไม่มีปุ่มนี้** (การออกจากเครื่องปัจจุบันใช้ "ออกจากระบบ" จากเมนูหลัก ไม่ใช่จากลิสต์นี้ กันสับสน)

**หัวหน้าจอ**
| องค์ประกอบ | ข้อความ |
|---|---|
| หัวข้อ | อุปกรณ์ที่เข้าสู่ระบบ |
| คำอธิบายรอง | รายการอุปกรณ์ที่กำลังเข้าใช้งานบัญชีของคุณอยู่ |
| ปุ่มท้ายรายการ (prominent) | ออกจากระบบทุกอุปกรณ์ |

**Empty/loading/error (มาตรฐาน design-system §2)**
| state | copy/UI |
|---|---|
| loading | skeleton shimmer 2-3 แถว (ไม่ใช่ spinner) |
| error (โหลดรายการไม่ได้) | ข้อความ "โหลดรายการอุปกรณ์ไม่สำเร็จ" + ปุ่ม "ลองใหม่" |
| empty (ทางทฤษฎีไม่ควรเกิด — อย่างน้อยต้องมี 1 คือเครื่องปัจจุบัน) | ไม่ต้องออกแบบ empty state พิเศษ (ถ้าเกิดจริงถือเป็น bug ไม่ใช่ UX state) |

**Interaction: "ออกจากอุปกรณ์นี้" (per-device, non-current — resolves open item #3)**
1. กดปุ่มที่ row → confirm dialog เล็ก (ไม่ทำทันทีแบบไม่ถาม เพราะย้อนกลับไม่ได้)
   - หัวข้อ: "ออกจากอุปกรณ์นี้?"
   - เนื้อหา: "อุปกรณ์นี้จะต้องเข้าสู่ระบบใหม่อีกครั้ง"
   - ปุ่ม: "ยกเลิก" (secondary) / "ออกจากอุปกรณ์นี้" (primary, สีแดง/destructive)
2. ยืนยัน → เรียก `POST /auth/logout { familyId }` → เอา row ออกจากลิสต์ทันที (optimistic) + toast "ออกจากอุปกรณ์แล้ว"
3. ถ้า fail → คืน row กลับ + toast แดง "ออกจากอุปกรณ์ไม่สำเร็จ กรุณาลองใหม่"

## 7. Edge states (cross-cutting)

| Edge case | พฤติกรรม UX |
|---|---|
| **Token access หมดอายุระหว่างใช้งาน** (15 นาที) | **เงียบ (silent refresh)** — client เรียก `/auth/refresh` เบื้องหลังอัตโนมัติเมื่อ access token ใกล้หมดอายุ/โดน 401 ครั้งแรก ผู้ใช้ไม่เห็นอะไรเลย ถ้า refresh สำเร็จ request เดิมทำต่อได้ (retry once) |
| **Refresh ใช้ไม่ได้แล้ว** (`401 INVALID_REFRESH` — หมดอายุจริง/ถูกเพิกถอน/reuse-detected) | เด้งออกไปหน้า `/login` ทันที + **toast สุภาพ ไม่กล่าวโทษ**: "เซสชันหมดอายุ กรุณาเข้าสู่ระบบอีกครั้ง" (ใช้ข้อความเดียวกันทุกสาเหตุ — ไม่แยกว่า "ถูกเพิกถอนเพราะสงสัยว่าโดนขโมย" เพื่อไม่รั่วข้อมูลให้ attacker และไม่ทำให้ผู้ใช้ตกใจเกินเหตุ) |
| **ถูก kick กลางการใช้งาน** (เช่น admin reset รหัส หรือกด "ออกจากระบบทุกอุปกรณ์" จากเครื่องอื่น) | รอบ request ถัดไปจะได้ 401 → พฤติกรรมเดียวกับแถวบน (เด้ง login + toast เดียวกัน) — ไม่ต้อง real-time push แจ้งเตือนใน MVP (WebSocket/SSE ไม่อยู่ใน scope F-001) |
| **Loading (ทุกฟอร์ม auth)** | ปุ่ม submit เข้า loading state (spinner เล็กในปุ่ม + disabled) — หน้าไม่ใช้ skeleton เพราะเป็น form action ไม่ใช่ data list (skeleton สำรองไว้กับ session list) |
| **Empty (session list)** | ดู §4 — ไม่ควรเกิดจริง |
| **Error เครือข่าย (ทุกฟอร์ม)** | banner แดงเหนือฟอร์ม + ปุ่ม "ลองใหม่" ตามมาตรฐาน design-system §2 |

## 8. Platform parity — web vs mobile (F-006)

| ประเด็น | Web | Mobile (F-006) |
|---|---|---|
| Layout สมัคร/ล็อกอิน | Centered card, max-width ~400px, full page | Full-screen form, safe-area aware |
| Refresh token | httpOnly cookie (ผู้ใช้ไม่เห็น/ไม่ต้องทำอะไร) | เก็บใน Keychain/Keystore (ผู้ใช้ไม่เห็น/ไม่ต้องทำอะไร) — **ไม่มีผลต่อ UI ที่ผู้ใช้เห็นเลย** ทั้งสอง platform แสดงฟอร์มเดียวกันทุกประการ |
| แสดง/ซ่อนรหัสผ่าน | ไอคอนตาในฟิลด์ (คลิก) | ไอคอนตาในฟิลด์ (แตะ) — เดียวกัน |
| Throttle countdown | banner เหนือฟอร์ม, นับถอยหลัง real-time | เดียวกันทุกประการ — component เดียวใน design-system, ใช้ร่วมกัน |
| Session list navigation | เมนู "บัญชีของฉัน" → "อุปกรณ์ที่เข้าสู่ระบบ" (sidebar/dropdown) | เมนู "ตั้งค่า" → "อุปกรณ์ที่เข้าสู่ระบบ" (bottom-nav/drawer ตาม F-006 IA) — เนื้อหาหน้าเดียวกันทุกประการ ต่างแค่ทางเข้า |
| Logout current device | ปุ่มใน user menu (dropdown มุมขวาบน) | ปุ่มใน settings screen | 
| Deep link หลัง logout-all | redirect `/login` | pop ไปหน้า login screen (clear nav stack ทั้งหมด กัน back กลับเข้าหน้าที่ต้อง auth) |
| Biometric unlock (Face ID/fingerprint) | ไม่มี (ไม่ใช่ web capability) | **นอก scope F-001** — ถ้า F-006 ต้องการ เป็น feature เพิ่มเติมที่ wrap ทับ refresh-token flow เดิม ไม่กระทบ contract นี้ |

**สรุปหลักการ parity:** ทุก copy, ทุก error state, ทุก confirm dialog — **ข้อความเดียวกันทุกตัวอักษร** ระหว่าง web/mobile (ผ่าน i18n key ชุดเดียวกัน, ดู [ui.md](./ui.md) §3) ต่างกันเฉพาะ layout/navigation shell ตามธรรมชาติของ platform.

"ความปลอดภัย" ให้ครอบคลุมทั้งรหัสผ่าน+เซสชัน — ตั้งชื่อเมนูใหม่)

### 9.3 States

| สถานการณ์ | ทริกเกอร์ | UI | ข้อความ |
|---|---|---|---|
| กำลังส่ง | submit กด | ปุ่ม loading state, disabled | "กำลังเปลี่ยนรหัสผ่าน..." |
| รหัสผ่านปัจจุบันผิด (`401 INVALID_CREDENTIALS`) | submit → server ตอบ | `ErrorBanner` เหนือฟอร์ม + border แดงที่ฟิลด์ "รหัสผ่านปัจจุบัน" เท่านั้น (ไม่ใช่ enumeration concern แบบ login — ผู้ใช้คือเจ้าของบัญชีอยู่แล้ว รู้ตัวเองว่าใครพิมพ์ผิด บอกตรงๆ ได้และช่วยแก้เร็วกว่า) | "รหัสผ่านปัจจุบันไม่ถูกต้อง" |
| รหัสผ่านใหม่สั้นเกิน (`422 PASSWORD_TOO_SHORT`) | submit → server ตอบ | border แดง + ข้อความใต้ฟิลด์ "รหัสผ่านใหม่" | "รหัสผ่านสั้นเกินไป ต้องมีอย่างน้อย 8 ตัวอักษร" |
| รหัสผ่านใหม่อยู่ในลิสต์รั่ว (`422 PASSWORD_BREACHED`) | submit → server ตอบ | border แดง + ข้อความใต้ฟิลด์ "รหัสผ่านใหม่" | "รหัสผ่านนี้ถูกใช้งานทั่วไปมาก ไม่ปลอดภัย ลองตั้งรหัสผ่านที่คาดเดายากขึ้น" |
| Throttle (`429`, ผิดรหัสปัจจุบันซ้ำๆ) | submit → server ตอบ | **`ThrottleBanner` เดียวกับ login** (§3.2) — เหลืองเตือน, นับถอยหลัง real-time, ฟอร์ม disabled จนครบเวลา | ใช้ copy ชุดเดียวกับ `auth.throttle.*` (§3.2/ui.md §3.3) — "รอสักครู่แล้วลองใหม่ · เหลือ {N} วินาที" ฯลฯ, **ห้ามคำว่า "ล็อก"/"ระงับ"** เหมือนเดิม (D-005 หลักการเดียวกันใช้ที่นี่ด้วย แม้จะเป็น brute-force ต่อ token ที่ล็อกอินอยู่ ก็ยังไม่ใช่ "บัญชีถูกระงับ") |
| Network/5xx | submit ล้มเหลว | `ErrorBanner` แดงเหนือฟอร์ม + ปุ่ม "ลองใหม่" | "เกิดข้อผิดพลาด กรุณาลองใหม่อีกครั้ง" |
| สำเร็จ (`200 { ok: true }`) | server ตอบสำเร็จ | เคลียร์ทั้งสองฟิลด์ + **toast สีเขียว** (ดู 9.4) | ดู 9.4 |

**เรื่อง "ยืนยันรหัสผ่านใหม่" (confirm-new-password field) — ตัดสินใจ: ไม่เพิ่ม**

Signup (§2) จงใจไม่มีฟิลด์ยืนยันรหัสผ่าน (ลดแรงเสียดทาน, ใช้ปุ่มตาแสดง/ซ่อนแทน) เพื่อความ
สม่ำเสมอ (consistency) จึงใช้หลักการเดียวกันที่นี่ — **ไม่เพิ่ม** ฟิลด์ยืนยัน แม้
change-password จะ "high-stakes" กว่าตรงที่พลาดแล้วกระทบเครื่องอื่นที่ถูกเพิกถอนออกไปด้วย
(ไม่ใช่แค่ตัวเอง). เหตุผลที่ยังไม่เพิ่ม:
- ปุ่มตาแสดง/ซ่อนรหัสผ่านที่มีอยู่แล้วทำหน้าที่ "ตรวจทานก่อน submit" ได้เทียบเท่าฟิลด์ยืนยัน
  โดยไม่เพิ่มขั้นตอนพิมพ์ซ้ำ — ผู้ใช้กดตาดูก่อนกดปุ่มได้เสมอ
- ถ้าพิมพ์รหัสใหม่ผิด (typo) โดยไม่ได้ตั้งใจ ผลคือ "ตั้งรหัสเป็นค่าที่พิมพ์ผิดนั้น" ซึ่งแก้ไขได้
  ทันที: ผู้ใช้ยังจำรหัสเดิม (current password) ได้แน่นอน (เพิ่งพิมพ์เอง) จึงกลับมาหน้านี้
  เปลี่ยนใหม่ได้ทันที ไม่ใช่ทางตันแบบ "ลืมรหัสผ่าน" (ต่างจากเคส signup ที่พลาดแล้วอาจไม่รู้ตัว
  จนกว่าจะ login ไม่ได้)
- เพิ่มฟิลด์ยืนยัน = เพิ่ม friction ให้ทุกคนเพื่อป้องกัน edge case ที่กู้คืนเองได้อยู่แล้ว —
  ไม่คุ้ม (skill thai-ux: ลดขั้นตอนที่ไม่จำเป็นสำหรับผู้ใช้ที่ไม่ใช่ tech-savvy)

**สรุป:** คงเหลือ 2 ฟิลด์ (ปัจจุบัน + ใหม่) + ปุ่มตา ไม่มีฟิลด์ที่ 3

### 9.4 Success messaging — "อุปกรณ์อื่นถูกออกจากระบบ" (สำคัญ)

เปลี่ยนรหัสผ่านสำเร็จ **ต้องแจ้งผู้ใช้ว่าอุปกรณ์อื่นถูกออกจากระบบ** — ใส่กรอบเป็นเรื่องดี
(security-positive) ไม่ใช่ผลข้างเคียงที่น่ากังวล และย้ำว่า **เครื่องที่ใช้อยู่นี้ยังคงล็อกอิน
ต่อได้ปกติ ไม่ต้อง login ใหม่**:

**Toast (สีเขียว, ปรากฏหลัง submit สำเร็จ, อยู่นานกว่า toast ปกติเล็กน้อยเพราะมีสองประโยค):**

> "เปลี่ยนรหัสผ่านแล้ว · อุปกรณ์อื่นถูกออกจากระบบเพื่อความปลอดภัย"

- ไม่ต้อง redirect ไปหน้า login (เครื่องปัจจุบันยังอยู่ในระบบตามสัญญา §2.7 — "keep current
  session" เมื่อ resolve ได้) — ผู้ใช้อยู่หน้า "ความปลอดภัย" ต่อ เห็นฟอร์มว่างพร้อมใช้ใหม่ได้
- ถ้าหน้านี้อยู่ติดกับ session list (§4, กรณีใช้ layout รวม) ให้ **refresh รายการ session ทันที
  หลัง toast** (optimistic หรือ re-fetch) เพื่อให้ผู้ใช้เห็นด้วยตาว่าเครื่องอื่นหายไปจริง —
  เสริมความน่าเชื่อถือของข้อความ toast โดยไม่ต้องพึ่งคำพูดอย่างเดียว
- **Edge case (server resolve current family ไม่ได้ — api-spec §2.7 "ไม่มี refresh token ที่
  resolve ได้"):** ผลจริงคือ revoke **ทุก**เครื่องรวมเครื่องปัจจุบัน → ผู้ใช้จะโดน 401 ที่
  request ถัดไปตามปกติ (edge state ทั่วไปใน §7 "ถูก kick กลางการใช้งาน") → เด้ง login พร้อม
  toast มาตรฐาน "เซสชันหมดอายุ กรุณาเข้าสู่ระบบอีกครั้ง" — **ไม่ต้องออกแบบ toast พิเศษแยกสำหรับ
  เคสนี้** (ผู้ใช้ทั่วไปแทบไม่เจอ เพราะเบราว์เซอร์/แอปจะมี refresh token ให้ resolve เกือบเสมอ)
  แต่ถ้า UX อยากกันความสับสน ("ทำไมจู่ๆ ต้อง login") ให้ frontend ส่ง `refreshToken` (mobile)
  ตาม §9.5 เสมอเพื่อลดโอกาสเจอ edge นี้ ไม่ใช่ทำอะไรเพิ่มฝั่ง UI

### 9.5 Data/transport notes (ไม่ต้องออกแบบ UI แต่ frontend ต้องรู้)

- Web: cookie `omni_rt` ส่งไปเองแบบ ambient + ต้องแนบ `X-CSRF-Token` (ตาม §0 ของ api-spec) —
  ไม่มี UI element เกี่ยวข้อง เป็นหน้าที่ frontend wiring
- Mobile: ควรส่ง `{ refreshToken }` (ของเครื่องปัจจุบัน) แนบไปด้วยในบอดี้ เพื่อให้ server resolve
  "current family" ได้แม่นยำ (มิฉะนั้นเข้า edge case revoke-all ใน 9.4) — ไม่กระทบ copy/flow
  ที่ผู้ใช้เห็น เป็นแค่ implementation detail ที่ต้องไม่ลืม

### 9.6 Platform parity

เหมือนหลักการ §8: **copy เดียวกันทุกตัวอักษร** ทั้ง web/mobile ผ่าน i18n key ชุดเดียวกัน
(`auth.change_password.*`, ui.md §3) ต่างกันเฉพาะ nav/ทางเข้า (F-006 IA):

| ประเด็น | Web | Mobile (F-006) |
|---|---|---|
| ทางเข้า | "บัญชีของฉัน" (dropdown มุมขวาบน) → "ความปลอดภัย" | "ตั้งค่า" → "ความปลอดภัย" (bottom-nav/drawer) |
| Layout | section ในหน้า `/settings/security`, การ์ดเดียวกับ/ติดกับ session list | screen เดียวกัน ต่างแค่ scroll แทน sidebar |

| toast placement | ตาม design-system (มุมขวาบน/ล่าง) | ด้านล่างจอ เหนือ safe-area |

## 10. Data dependencies confirmed (ไม่มีของค้าง)

ทุกฟิลด์ที่ใช้ในจอข้างต้นมาจาก contract ที่ lock แล้วใน [api-spec.md](./api-spec.md):
- Signup: `{ email, password }` → `{ userId, email, verified }` / errors §2.1
- Login: `{ email, password, deviceId? }` → `{ accessToken, refreshToken, expiresIn, tokenType }` / `Retry-After` header §2.2
- Sessions: `{ sessions: [{ familyId, deviceId, createdAt, lastUsedAt, current }] }` §2.6 — ใช้ตรงกับ layout §4 ทุกฟิลด์ (ไม่ต้องขอเพิ่ม)
- Logout (per-device): optional `{ familyId }` body §2.6 note — ตามที่ resolve ไว้ open item #3
- Change password: `{ currentPassword, newPassword }` → `{ ok: true }` / errors `401`/`422`/`429`
  §2.7 — ใช้ตรงกับ §9 ทุกฟิลด์ (ไม่ต้องขอเพิ่ม); "current family spared" resolve จาก presented
  refresh token ฝั่ง server ล้วนๆ ไม่มี field ใน response ที่ UI ต้องอ่าน (ไม่มี "familyId ที่ถูก
  spare" ส่งกลับมา — ไม่จำเป็นสำหรับ copy ใน §9.4)

ไม่มี field ที่ UX ต้องการแต่ backend ยังไม่มี — **ไม่มี BLOCKED ต่อ backend-api ในเอกสารนี้**

```
┌──────────────────────────────────────┐
│  เปลี่ยนรหัสผ่าน                        │
│                                        │
│  รหัสผ่านปัจจุบัน                       │
│  ┌──────────────────────────────────┐ │
│  │                                👁 │ │
│  └──────────────────────────────────┘ │
│                                        │
│  รหัสผ่านใหม่                          │
│  ┌──────────────────────────────────┐ │
│  │ อย่างน้อย 8 ตัวอักษร          👁 │ │
│  └──────────────────────────────────┘ │
│  ใช้ตัวอักษร ตัวเลข หรือวลีที่จำง่าย —    │
│  ไม่ต้องมีอักขระพิเศษก็ได้                │
│                                        │
│  ┌──────────────────────────────────┐ │
│  │         เปลี่ยนรหัสผ่าน             │ │  ← primary
│  └──────────────────────────────────┘ │
└──────────────────────────────────────┘
```

หมายเหตุ: ไม่มีฟิลด์ "ยืนยันรหัสผ่านใหม่" ที่ 3 (ตัดสินใจแล้ว §9.3) — 2 ฟิลด์ + ปุ่มตาต่อฟิลด์เท่านั้น.

**Error state (รหัสผ่านปัจจุบันผิด):**

```
│  ⚠ รหัสผ่านปัจจุบันไม่ถูกต้อง             │  ← ErrorBanner เหนือฟอร์ม
│  รหัสผ่านปัจจุบัน                       │
│  ┌──────────────────────────────────┐ │
│  │                                👁 │ │  ← border แดง (เฉพาะฟิลด์นี้)
│  └──────────────────────────────────┘ │
```

**Throttle state:** ใช้ `ThrottleBanner` เดียวกับ 11.3 ทุกประการ (แปะเหนือฟอร์มนี้แทน).

**Success toast (เขียว, มุมขวาบน web / เหนือ safe-area ล่าง mobile):**

```
┌────────────────────────────────────────────┐
│ ✓ เปลี่ยนรหัสผ่านแล้ว · อุปกรณ์อื่นถูกออกจาก   │
│   ระบบเพื่อความปลอดภัย                       │
└────────────────────────────────────────────┘
```

หลัง toast: ฟิลด์เคลียร์, ถ้าอยู่หน้าเดียวกับ session list (11.5) ให้ list refresh ทันทีให้เห็นว่า
เครื่องอื่นหายไปจริง (§9.4).

### 11.7 Confirm dialogs

**ออกจากอุปกรณ์นี้ (per-device, non-current):**

```
        ┌────────────────────────────────┐
        │  ออกจากอุปกรณ์นี้?               │
        │                                  │
        │  อุปกรณ์นี้จะต้องเข้าสู่ระบบใหม่     │
        │  อีกครั้ง                         │
        │                                  │
        │     [ ยกเลิก ]  [ ออกจากอุปกรณ์นี้ ]│
        │       secondary      destructive │
        │      (default focus)    แดง      │
        └────────────────────────────────┘
```

**ออกจากระบบทุกอุปกรณ์ (destructive, กระทบทุกเครื่องรวมเครื่องนี้):**

```
        ┌────────────────────────────────┐
        │  ออกจากระบบทุกอุปกรณ์?           │
        │                                  │
        │  ทุกอุปกรณ์ที่เข้าสู่ระบบอยู่         │
        │  (รวมเครื่องนี้) จะถูกออกจากระบบ    │
        │  ทันที และต้องเข้าสู่ระบบใหม่ทุกเครื่อง│
        │                                  │
        │   [ ยกเลิก ]  [ ออกจากระบบทุกอุปกรณ์ ]│
        │    secondary        destructive  │
        │   (default focus)      แดง       │
        └────────────────────────────────┘
```

> Focus: ปุ่ม "ยกเลิก" เป็น default focus เสมอสำหรับ variant destructive (กัน Enter พลาด —
> ตาม accessibility notes ui.md §6).

**Toast หลัง confirm สำเร็จ (ตัวอย่าง — ทั้งสองแบบ, เขียว):**

```
┌───────────────────────────────┐      ┌───────────────────────────────┐
│ ✓ ออกจากอุปกรณ์แล้ว              │      │ ✓ ออกจากระบบทุกอุปกรณ์แล้ว        │
└───────────────────────────────┘      └───────────────────────────────┘
      (per-device)                            (logout-all → redirect /login)
```

**Web:** modal กลางจอ (มี overlay มืดคลุมหลัง). **Mobile:** bottom sheet เลื่อนขึ้นจากล่างจอ —
เนื้อหา/ปุ่ม/ลำดับ default focus เหมือนกันทุกตัวอักษร ต่างแค่ container/animation (§8).
